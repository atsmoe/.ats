2026-6-29 — V2.5.1：滚动性能修复 · 重复处理器清理

## 1. 诊断过程

  V2.5.0 上线后用户反馈页面浏览比之前卡顿。三个 Explore agent 并行排查：
  - Agent A：追踪正常页面加载路径（main.js → timeline-ui.js → portal-transition.js → data-loader.js → virtual-timeline.js）
  - Agent B：检查构建产物（bundle.js 体积、图片输出、HTML script 加载顺序、CSS 动画开销）
  - Agent C：审计滚动渲染性能（scroll handler 效率、_renderRange 节点管理、RAF 调度、合成层压力）

## 2. 根因：重复滚动处理器

  发现 `window` 上注册了两个内容完全重复的 scroll 事件监听器：

  | 位置 | 节流 | 每次滚动执行 |
  |------|------|-------------|
  | `virtual-timeline.js:346` | 无节流 | `VirtualTimeline.update()` + `updateEraNavHighlight()` + `updateBackToTop()` |
  | `timeline-ui.js:446` | RAF 节流 | 同上三件事 |

  每次滚动事件触发时：
  - virtual-timeline.js 同步执行 `updateEraNavHighlight()` → `VirtualTimeline.items.filter(item => item.type === 'era-header')` 遍历全部 876 items
  - timeline-ui.js 在下一 RAF 帧再次执行相同逻辑
  - 每帧滚动 = `updateEraNavHighlight()` 执行两次 = **876 × 2 = 1752 次迭代** + 两次 `querySelectorAll` + 两次 `forEach` DOM 操作

  V2.5.0 自身的 JS 改动全都不在正常滚动路径上（portal 逻辑只在带 hash 时执行，fetchWithRetry 成功路径零开销，图片渲染仅在弹窗打开时触发），但预存的重复处理器问题被充分暴露。

## 3. 修复方案

  ### 3.1 移除 virtual-timeline.js 的 scroll handler
  - 删除第 346-352 行的 `window.addEventListener('scroll', ...)` 
  - 保留 resize handler 不变（timeline-ui.js 没有 resize handler）
  - timeline-ui.js 的 RAF 节流版本独自承担滚动处理

  ### 3.2 清除 window.renderEvents 死代码
  - `window.renderEvents`（第 493-515 行）内部调用 `getBranchEvents(branch)`，该函数是 timeline-ui.js 的 ES module 导出，不在全局作用域 → 调用即抛 `ReferenceError`
  - V2.5.0 已移除 portal 动画完成回调中的唯一调用点
  - 将 `renderEventsDirect()` 保留为 `VirtualTimeline._renderEventsDirect` 备未来使用

## 4. 附带发现（未修复，留待后续）

  - `_renderRange()` 每次范围变化时通过 `querySelectorAll` + `el.remove()` 销毁全部 ~15 个可见节点，再从 `_buildNode()` 重建。`nodePool` Map 已声明但从未用于节点回收——死数据结构
  - `_renderRange()` 在渲染前 O(n) 扫描 range.start 前所有 item 以计算事件索引（交替左/右侧）
  - 每个 `.event-card` 使用 `backdrop-filter: blur(8px)`——15 个可见卡片 = 15 个背景滤镜合成层，在低端设备上造成 GPU 压力
  - 3 处 `transition: all` 可收紧为具体属性

## 5. 验证结果

  - 构建通过，无新增 warning
  - dist 产物确认：scroll handler 已从 virtual-timeline.js 移除，`window.renderEvents` 不存在
  - dev server 正常启动，各页面可访问
  - 滚动时 `VirtualTimeline.update()` 仅通过 RAF 节流调用一次

## 4. 模板架构

- 基础布局 base.njk 定义 <html> 骨架、#portal-arrival 内联遮罩（<body> 第一个元素）、导航、页脚、背景 canvas、虫洞 canvas、JS 引用
- 11ty front matter 声明 permalink、title、bodyClass、pageType、worldId、cssFiles
- CSS 加载：{% if eleventy.env.runMode === "serve" %} 用 linkCSS shortcode 输出 <link> 标签，否则用 inlineCSS shortcode 内联
- 数据内联：base.njk 中根据 worldId 条件调用 worldDataScript filter，输出 <script>window.__WORLD_DATA__ = {...};</script>
- 导航高亮：nav.njk 中通过 {% if worldId === 'arknights' %} 等判断加 .active 类

## 5. JS 模块化细节

- 源码使用 ES modules（import/export）
- esbuild 打包为 IIFE 格式 dist/js/bundle.js（62KB），virtual-timeline.js 单独复制
- BackgroundManager 从 hardcode getElementById 改为 init(canvasId, videoId, imageId) 模式
- ParticleBackground 的星系绘制中，detailGalaxy/hoveredGalaxy 通过 window.__detailGalaxy/__hoveredGalaxy 与 star-map.js 通信
- main.js 入口根据 body.dataset.page 动态 import 对应模块（star-map / world / about 三条路径）
- 跨世界引用：.cross-world-link 的全局 click 委托 → playPortalOutgoing → 虫洞动画 → window.location.href 跳转
- 到达侧：initPortalArrival 检测 location.hash → 在数据中搜索目标事件 → 直接 VirtualTimeline.load() + estimateScrollTopByEventId() 预估定位 → update() + remeasure 校正 → 反向收束动画 → dismissPortal（无需全量渲染，无二次加载）

## 6. 构建与命令

- npm run dev：启动 11ty 开发服务器（localhost:9000），支持热更新
- npm run build：11ty 构建 HTML + esbuild 打包 JS + 复制 CSS → dist/
- file:// 协议：dist/ 下 HTML 直接双击打开（JSON 数据已内联，无需 fetch）

## 7. 传送门过渡（Portal Transition）

出发侧：
  1. 用户点击 .cross-world-link（e.preventDefault）
  2. 以点击坐标为圆心，Canvas 涡旋粒子向外螺旋扩散
  3. 光晕完全不透明时 → window.location.href = targetUrl

到达侧：
  1. #portal-arrival 内联 div 从 HTML 解析第一刻遮住全屏（#060a14 深空底色）
  2. 读取 window.__WORLD_DATA__（内联数据，瞬时可用）
  3. 解析 location.hash 得到目标事件 ID
  4. 临时关闭虚拟滚动 → 全量渲染所有节点
  5. scrollTo + requestAnimationFrame 等待目标节点
  6. 播放反向收束动画（Canvas 粒子 + 光晕收缩到目标位置，~400ms）
  7. 移除遮罩，恢复虚拟滚动
